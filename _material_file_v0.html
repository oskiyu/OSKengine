Archivo de descripción de material (revisión 0).

En esta versión de la especificación, la información del material se divide entre el archivo de material y el archivo de shader.

Atención
Esta versión de la especificación únicamente permite usar bindings del tipo SAMPLER, CUBEMAP o UNIFORM BUFFER. No soporta STORAGE IMAGES, STORAGE BUFFERS o arrays de samplers.
Parámetros
file_typeTipo de archivo JSON. Dependiendo del tipo de material, será uno u otro. Tipos de material.
spec_verVersión de la especificación del material.
nameNombre del material (único).
shader_fileRuta de acceso al archivo JSON de shader usado por el material.
layoutEstructura de datos que define el layout del material, indicando sus bindings (buffers, texturas, etc...) y los push contsants. Layout
vertex_type[condicional] Nombre del tipo de vértice que se va a usar en este material (para materiales gráficos).
config[opcional] Estructura de datos con configuración adicional.
polygon_mode[opcional] Tipo de polígono usado, en caso de tratarse de un material gráfico. Modo de renderizado de polígonos.
Precondición
file_type debe coincidir con uno de los tipos presentados en Tipos de material..
polygon_mode debe coincidir con uno de los tipos presentados en Modo de renderizado de polígonos..


Precondición
spec_ver debe ser 0 para que se aplique esta especificación.
name debe ser único entre todos los materiales cargados en el motor en un momento dado.
El shader indicado por shader_file debe ser compatible con el layout definido en este archivo.


Precondición
Si vertex_type está presente, entonces file_type debe ser "MATERIAL".
vertex_type debe coincidir con el nombre del tipo de vértices en el código C++.


Precondición
Si polygon_mode está presente, entonces file_type debe ser "MATERIAL".

Layout

Parámetros
push_constantsInformación sobre los push constants del material. Contiene un array de Push Constant.
slotsInformación sobre los slots del material. Contiene un array de Slot.
Nota
Incluso si se dispone de un único push constant, este debe estar introducido dentro de un array de JSON.
Si no se dispone de ningún push constant, debe existir un array vacío en el archivo JSON.
Incluso si se dispone de un único slot, este debe estar introducido dentro de un array de JSON.
Si no se dispone de ningún slot, debe existir un array vacío en el archivo JSON.

Push Constant

Parámetros
nameNombre del push constant.
sizeTamaño del push constant, en bytes.
Precondición
name debe ser único entre todos los push constants de este material.
size debe ser mayor que 0.
size debe cumplir con los límites de tamaño de push constant de la plataforma de destino.

Slot

Parámetros
nameNombre del slot.
bindingsArray de Binding.
Precondición
name debe ser único entre todos los slots de este material.

Binding

Parámetros
nameNombre del binding.
typeTipo del binding.
Precondición
name debe ser único entre todos los bindings del slot al que pertenece.
El tipo de binding debe ser SAMPLER, CUBEMAP o UNIFORM BUFFER.
Si el tipo de binding es SAMPLER, type debe ser "IMAGE".
Si el tipo de binding es CUBEMAP, type debe ser "CUBEMAP".
Si el tipo de binding es UNIFORM BUFFER, type debe ser "UNIFORM".

Tipos de material.

Existen tres tipos de material, dependiendo del tipo de pipeline y shaders que aceptan:

  • Material de rasterizado, representado por "MATERIAL".
  • Material de trazado de rayos, representado por "MATERIAL_RT".
  • Material de computación, representado por "MATERIAL_COMPUTE".

Modo de renderizado de polígonos.

Existen tres maneras de renderizar geometría:

  • Renderizado tradicional de triángulos, representado por "FILL".
  • Renderizado de triángulos en wideframe, representado por "TRIANGLE_WIDEFRAME".
  • Renderizado de líneas (no triángulos), representado por "LINE".

Modo de depth testing.

Existen tres maneras de configurar el depth testing:

  • Desactivado, representado por "none".
  • Sólo lectura: se comprueba la geometría, pero no se sobreescribe la imagen de profundidad. Representado por "read".
  • Comportamiento normal: se comprueba la geometría, y se sobreescribe la imagen de profundidad si es necesario. Representado por "read/write".

Por defecto, está activado en modo read / write.

Modo de culling.

Existen tres maneras de configurar el depth testing:

  • Desactivado, representado por "none".
  • Front-face culling, representado por "front".
  • Back-face culling, representado por "back".

Por defecto, está activado en modo back.
